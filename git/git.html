Aprendendo Git!

Para que surgiu o git?

A ferramenta Git surgiu para que projetos fossem commitados, ou seja ter verções que possamos ter de qualquer aplicação.

Seus projetos vão sofrendo mudanças, o Professor Guanabara do Youtube disse em uma de suas aulas que um projeto nunca está finalizado, sempre tem o que melhorar ou adicionar.

E quando vamos melhorando ou adicionando funcionalidades a nossas aplicações podemos ter alguns imprevistos ou até pedidos de clientes que querem voltar a maneira entiga do projeto.

O git surgiu justamente para resolver esse problema, com comando simples podemos voltar a como projeto era, se não tivessemos esse programa, teriamos muitos problemas. Quando um projeto é pequeno, ainda dá pra resolver umas mudancinhas, mas quando temos uma grande aplicação temos que ter esse recurso para termos mais praticidade e eficiencia.

Comandos

Se apresentando

Para começar vamos adicionar algumas informações suas no seu git, pois quando estivermos trabalhando com alguem em um projeto, precisamos saber com quem estamos falando. O primeiro comando no seu git então é o git confg -global usar.name "" e dentro das aspas vamos digitar nosso nome. Depois vamos usar o git confg --global user.email "" e novamente destro das aspas coloque seu e-mail de contato.

Para iniciar

No visual studio code vamos da nosso primeiro comando para inicializarmos o git. Para abrir o terminal usamos as teclas ctrl j. E no terminal vamos digitar o comando git init. Nessa hora vai aparecer uma mensajgem informando que o repositorio git foi criado e irá aparecer uma pasta com nomeada git na pasta que vecê está ultilizando.

Como vê pastas ocultas no gerenciador de arquivos

quando fizer o comando acima pode ser que vecê não veja nada na pasta, isso acontece por que não está configurado para mostrar pastas ocultas. Para mostrar é simples, vá em exibir, depois em opções no seu gerenciador de arquivos e da você seleciona mode de exibição, descendo um pouco a barra irá achar a opção mostrar aquivos, pastas e unidades ocultas.

Preparar para commit

Para usar como exemplo vamos criar duas páginas, index.html e script.js. No terminal vamos usar o comando git status. Esse comando mostrará todos os arquivos da sua aplicação, e se eles estão commitados, preparados para commitar ou não preparados para commitar. No caso como esses arquivos foram recem criados, com o comando git status mostrará que eles não estão preparados para commit, e para podermos preparalos temos algumas formas.

Podemos usar um comando para preparar um de cada vez. Dessa forma: git add index.html. Se quisermos removermos arquivos preparados para commitar, podemos usar o comando git rm --cached index.html. Já se quisermos preparar todos os arquivos, usamos o comando git add .

Primeiro comit

Com os arquivos preparados para commit usaremos o comando git commit -m "" (dentro das aspas vamos adicionar uma descrição). Nesse exemplo vamos colocar exatamente assim: git commit -m "Primeira verção com index.html e script.js"

Pronto! Vizemos nosso primeiro commit, para visualizarmos todos os commits podemos usar o comando git log.

Ignore Aquivos

Alguns arquivos da sua aplicação podem ter algo que você não queira dexar visivel. Senhas, Dados pessoais, entre outros motivos. Para não commitar alguma pasta, basta não preparar o arquivo né? Não! No dia a dia vamos trabalhar com muitos arquivos, então fica inviavel preparar de um por um, o mais fácil é usar um recurso do git para colocarmos os arquivos que não queresmos commitar. Pois tudo que vamos commitar iremos deixar no github, e não é uma boa prática deixar dados sensiveis assim.

Então a forma mais simples de resulver esse problema é criarmos um arquivo nomeado assim: .gitignore Em seguida dentro desse arquivo escrevermos o nome dos arquivos que não vamos commitar. Dessa forma reolvemos muito simplesmente o problema.

Branch

Um conceito muito importante é o de branch, note que quando você abro o terminal com o bash do git, vai aparecer um nome entre colchetes" (master) ". Normalmente uma aplicação tem dois ambientes, o de produção que é o ambiente que o cliente ta usando. Vamos colocar como exemplo um app, o que o cliente fica usando fica no branch master. O outro ambiente é o de desenvolvimento, que vai ficar o código que está sendo testado, um ambiente onde os desenvolvedores estão usando constantimente. O código que será trabalhado até ser enviado para aprodução.

Vamos supor que recebemos uma tarefa de adicionar uma funcionalidade de cadastro no nosso app, Não é interessante mandar para o ambiente de produção(do cliente) funcionalidades que não sabemos se tem algum erro ou coisa do tipo. Então temos que ter outro ambiente para trabalharmos o código até termos certeza que a funcionalidade nova não irá quebrar a aplicação.

Criando uma branch development

Muuito simples, para criar uma branch development é só escrever o comando git branch development. Você não notará nada de diferente, mas se usar o comando git branch o terminal mostrará todas as branchs existentes e qual que está em uso.

Pensem em branch como se fossem pequenas pastas dentro do nosso repositorio. Para podermos mudar de branch usamos o comando git checkout development. Nesse caso foi o development, mas depende para qual ambiente você quer ir. Se quiser voltar para o master é só colocar o mesmo comando com o nome master git checkout master

Vamos usar como se fosse um cadastro de cliente. No terminal vamos criar essa branck com o comando git branch cadastro-cliente Em seguida vamos vê os ambientes que temos com o comando git branch, então vamos entrar nesse ambiente com o comando git checkout cadastro-cliente. No exemplo vamos criar uma pasta cadastro-cliente e dentro dessa pasta um arquivo com mesmo nome.

Se usarmos o comando git status veremos que temos a pasta criada não preparada para commitar, então vamos preparar ela e tambem vamos commitar. Os comandos são git add . para preparar todo que tiver que preparar e git commit -m "cadastro de cliente adicionado"

Pronto, agora para você como funciona essa mudança de ambientes na prática, vamos mudar de branch mais uma vez com o comando git checkout development. Se olhar nos seus arquivos agora, vai notar que a pasta não está mais lá. Agora vamos supor que terminamos a funcionalidade de cadastro de clientes e queremos colocar no brnach do development. É simples vamos entrar no ambiente que irá receber o outro, nesse caso o git checkout development e usar o comando git merge cadastro-cliente. Pronto! Já está integrado com a branch development. Agora vamos supor que ta tudo tudo testado e queremos mandar para o cliente. Vamos entrar no git checkout master e aqui vamos usar o comando git merge development
